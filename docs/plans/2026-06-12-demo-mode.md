# Demo Mode — full lifecycle without Stripe/Resend keys

**Date:** 2026-06-12
**Goal:** A stakeholder can watch the entire MosRev concept work live — trial countdown,
day-23 reminder email, paywall lockout, checkout, subscription activation, day-37
win-back, workflow decision/invite emails — on a deployment with **no Stripe and no
Resend keys**. Driven by one env var: `DEMO_MODE=true`.

## Principles

- **Real keys always win.** Demo billing activates only when `STRIPE_SECRET_KEY` is
  unset; demo email only when `RESEND_API_KEY`/`MOSREV_EMAIL_FROM` are unset. Setting
  real keys silently disables the corresponding simulation — no code change to go live.
- **Inert outside demo mode.** Every demo endpoint 404s unless its demo gate is on.
  Without `DEMO_MODE`, behavior is byte-for-byte today's behavior.
- **Same code paths.** Demo mode doesn't fork the paywall or the email senders — it
  makes `isBillingEnabled()` true (so the real trial/paywall logic runs) and swaps the
  email transport for a DB-backed outbox. What the audience sees *is* the product logic.

## Contracts

### `src/lib/demo.ts` (new)
```ts
isDemoMode(): boolean      // DEMO_MODE?.trim().toLowerCase() === "true"
isDemoBilling(): boolean   // isDemoMode() && !isStripeConfigured()
isBillingEnabled(): boolean// isStripeConfigured() || isDemoBilling()
isDemoEmail(): boolean     // isDemoMode() && !(RESEND_API_KEY && MOSREV_EMAIL_FROM)  (reads env directly — no import of email.ts, avoids cycle)
```
`demo.ts` may import `stripe.ts` only.

### Gate swaps (`isStripeConfigured()` → `isBillingEnabled()`)
- `src/lib/subscription.ts` — `isSubscriptionActive` first line.
- `src/lib/require-subscription.ts` — `isOrgSubscribed` first line.
- `src/app/api/auth/me/route.ts` — trial countdown gate; also add `demo: isDemoMode()`
  to the returned `user` object (Navbar consumes it).
- `src/app/api/cron/trial-emails/route.ts` — stripe-unconfigured skip becomes
  billing-disabled skip.
- `src/app/billing/page.tsx` — `configured` becomes `billingEnabled`.

### Email capture
- New Prisma model:
  ```prisma
  model DemoEmail {
    id        String   @id @default(cuid())
    to        String
    subject   String
    text      String
    html      String?
    createdAt DateTime @default(now())
    @@index([createdAt])
  }
  ```
  Hand-written migration `prisma/migrations/20260612000000_add_demo_email/`. No RLS —
  global demo-only table; reads/writes go through `prismaAdmin`. (rls-roles.sql
  `ALTER DEFAULT PRIVILEGES` already grants both roles access to new tables.)
- `src/lib/notifications/email.ts`: at top of `sendEmail`, when `isDemoEmail()` →
  `prismaAdmin.demoEmail.create(...)` (join array recipients with ", "), return
  `{ ok: true, id }`. `isEmailConfigured()` returns true when `isDemoEmail()`.

### Trial email sweep extraction
- New `src/lib/trial-email-sweep.ts`: `runTrialEmailSweep()` — the body of today's cron
  route (claim-then-send semantics preserved verbatim), returning
  `{ reminders, winbacks, failures }`. Cron route keeps auth + gates and delegates.
  Demo controls call the same function.

### Demo billing endpoints (all: 404 unless `isDemoBilling()`; session + org + owner/admin + CSRF)
- `POST /api/billing/demo-checkout` — marks the active org subscribed:
  `subscriptionStatus="active"`, `stripeSubscriptionId="demo_sub_"+randomUUID()`,
  `stripeCustomerId="demo_cus_"+randomUUID()` if null, `currentPeriodEnd=+30d`,
  `lastStripeEventAt=now`. Via `prismaAdmin` (same cross-context rule as webhooks).
- `POST /api/billing/demo-controls` — zod body `{ action }`:
  - `day23`: org `createdAt=now-23d`, back to default trial (`subscriptionStatus="trialing"`, `stripeSubscriptionId=null`, `currentPeriodEnd=null`), `trialReminderSentAt=null`
  - `day31`: same reset but `createdAt=now-31d` (trial expired → paywall locks)
  - `day37`: same reset but `createdAt=now-37d`, `trialWinbackSentAt=null`
  - `reset`: `createdAt=now`, fresh trial, both sent-stamps cleared
  - `sweep`: run `runTrialEmailSweep()`, return its counts
- `src/app/api/billing/checkout/route.ts`: after CSRF, when `isDemoBilling()` return
  `{ url: "/billing/demo-checkout" }` before the Stripe 503 check.

- `src/app/billing/DemoControls.tsx` — "Demo time machine" card on /billing, only when

- `src/app/demo/outbox/page.tsx` — `notFound()` unless `isDemoMode()`; latest 50
  DemoEmails; per-row expand with sandboxed `<iframe srcDoc>` HTML preview (no scripts)
  and text fallback.
- `src/components/Navbar.tsx` — when `me.user.demo`: "Outbox" nav item + amber "Demo" chip.
- `src/app/layout.tsx` — global thin amber banner when `isDemoMode()`:
  "Demo mode — payments and emails are simulated."

### Seed (demo data)
`prisma/seed.ts`: when `DEMO_MODE=true`, additionally create 4 idempotent workflows in
the seeded org mirroring exactly how the app creates them (read
`src/app/api/workflows/route.ts` + approve route first — approval rows, currentStep,
status transitions, audit-log action strings must match):
1. Expense Approval — fully **approved** (both steps, notes, a comment, audit trail) → receipt downloadable
2. Zakat Distribution — **in_progress** (step 1 of 3 approved)
3. Donation Acknowledgment — **pending**
4. Expense Approval — **rejected** at step 2 with note
