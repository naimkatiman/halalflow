# Plan: Malay/English language toggle + dark/light mode

Date: 2026-06-14
Branch: `feat/i18n-dark-mode`
Status: in progress

## Goal

Add a user-controllable language switch (Malay `ms` / English `en`) and a dark/light
theme toggle to MosRev. SSR-correct, no flash, persisted across reloads. First pass
wires the full mechanism and applies it to the app chrome + key surfaces; the rest of
the app rolls in as follow-up commits.

## Decisions (confirmed with owner)

- **Scope:** foundation + chrome first. Mechanism applied to Navbar, footer, demo
  banner, landing page, login, register, dashboard. Remaining ~24 pages are a
  follow-up rollout and remain fully usable in the meantime.
- **Persistence:** cookies (`locale`, `theme`). SSR-aware, no DB change.
- **i18n engine:** typed dictionary + React context + `useT()` hook. No routing
  changes, no middleware (AGENTS.md warns this is a customized Next.js 16).
- **Default language:** auto-detect from `Accept-Language` on first visit, then the
  cookie wins.

## Verified framework facts (Next.js 16.2.6 / Tailwind 4.2.2)

- `cookies()` and `headers()` are **async** (`await cookies()`, `await headers()`).
  Confirmed in `node_modules/next/dist/docs`.
- Reading cookies/headers in the root layout opts the tree into dynamic rendering.
  Acceptable: app runs on a long-lived Node server (`next start`) and is already
  almost entirely auth/Prisma-dynamic.
- Cookies set during Server Component render are not allowed; mutate via client
  (`document.cookie`) or a Server Action/Route Handler. We set client-side
  (non-sensitive UI prefs, needs JS access for instant theme toggle).
- Tailwind v4 class-based dark mode: override the default `prefers-color-scheme`
  variant with `@custom-variant dark (&:where(.dark, .dark *));` in `globals.css`.
  `@custom-variant` is a recognized at-rule in 4.2.2.

## Architecture

Three layers.

1. **Server resolution** тАФ `src/lib/i18n/index.ts`
   - `Locale = 'en' | 'ms'`, `DEFAULT_LOCALE`, cookie names.
   - `getLocale()`: read `locale` cookie; fallback parse `Accept-Language` -> `ms` if a
     Malay tag (`ms`/`ms-MY`) outranks English, else `en`.
   - `getTheme()`: read `theme` cookie; fallback `'light'` (system pref handled by the
     no-flash script, since the server cannot read `prefers-color-scheme`).
   - `parseAcceptLanguage(header)`: pure, unit-tested.

2. **Dictionary + context (i18n engine)**
   - `src/lib/i18n/dictionaries/en.ts` (source-of-truth shape) + `ms.ts`.
   - `Dictionary = typeof en`. `ms.ts` is typed `: Dictionary` so missing keys fail
     the build.
   - `src/lib/i18n/provider.tsx` тАФ client `LocaleProvider` (seeded with server locale)
     + `useT()` returning a `t(key)` lookup against the active dictionary. Both
     dictionaries ship to client (small); switch is in-memory.
   - Server components in scope read strings directly from the imported dictionary by
     locale (no hook on the server).

3. **Persistence + toggles (client)**
   - `src/components/ThemeToggle.tsx` тАФ sun/moon (Phosphor). Toggles `.dark` on
     `<html>` instantly, writes `theme` cookie (1y, `path=/`, `SameSite=Lax`,
     non-httpOnly). Pure CSS, no reload.
   - `src/components/LanguageToggle.tsx` тАФ EN/MS pill. Writes `locale` cookie, flips
     provider locale instantly for client components, calls `router.refresh()` so
     server components re-render translated.
   - `src/components/ThemeScript.tsx` тАФ tiny inline blocking script: if no `theme`
     cookie, read `prefers-color-scheme` and set `.dark` before paint. No-op when the
     cookie exists (server already set the class). `suppressHydrationWarning` already
     on `<html>`.

## Dark-mode color strategy

- One line in `globals.css` switches `dark:` to class-based.
- `.dark { --background; --foreground; --border; --muted; }` overrides so the body and
  any token-driven surface flip for free. Add a couple of surface tokens
  (`--surface`, `--surface-muted`) used by chrome.
- In-scope components: convert literal light-only classes (`bg-white`, `text-zinc-900`,
  `border-zinc-200/50`) by adding `dark:` variants. Full-app color conversion is the
  follow-up rollout.

## Commit phasing (one concern each, <=15 files)

1. **Foundation** тАФ i18n lib + dictionaries + providers + `globals.css` dark
   tokens/variant + `layout.tsx` wiring + `ThemeToggle` + `LanguageToggle` +
   `ThemeScript` + `getLocale` unit test.
2. **Chrome** тАФ `Navbar.tsx` + footer (in `layout.tsx`) + demo banner translated +
   dark variants; mount the two toggles in the navbar.
3. **Landing** тАФ `src/app/page.tsx` translated + dark.
4. **Auth + dashboard** тАФ `login`, `register`, `dashboard` translated + dark.

## Verification

- `npm run build` green after each phase (type-check + compile). `ms.ts: Dictionary`
  guarantees key parity at build time.
- `npm run test` тАФ `parseAcceptLanguage` unit tests (vitest already configured).
- Manual: theme persists + no flash on reload; language toggle swaps chrome+landing
  and persists; Malay `Accept-Language` first-visit renders `ms`.
- Final: parallel adversarial review (correctness, dark coverage, i18n key integrity,
  a11y, Next 16 API usage).

## Risks / non-goals

- Non-goal: translating or theming the ~24 out-of-scope pages this pass. They stay
  usable (current strings/light theme) until the rollout reaches them.
- Risk: forcing dynamic rendering app-wide. Accepted (Node server, already dynamic).
- Not pushing to `main` (protected). Work lands on `feat/i18n-dark-mode`; integration
  decided with owner at the end.
